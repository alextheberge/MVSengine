You are preparing this project for `mvs-manager`, a multidimensional version manager that tracks:

ARCH.FEAT.PROT-CONT

Your job is to make this repository scannable and verifiable by `mvs-manager` without changing behavior unless required.

Work instructions:

1. Initialize or refresh the manifest.
   - If `mvs.json` does not exist, create it by running:
     `mvs-manager generate --root . --manifest mvs.json --context <context>`
   - Choose a short runtime context that matches the app, for example:
     `cli`, `web`, `server`, `edge`, `edge.mobile`, `desktop`
   - If `mvs.json` already exists, preserve its intent and regenerate it instead of replacing it by hand.

2. Add MVS comment decorators to real source files.
   - `mvs-manager` scans these file types:
     `.rs`, `.ts`, `.tsx`, `.js`, `.jsx`, `.go`, `.py`, `.java`, `.kt`, `.cs`, `.php`, `.rb`, `.swift`, `.lua`, `.luau`
   - It ignores these directories:
     `.git`, `node_modules`, `dist`, `build`, `target`, `vendor`, `.next`, `tests`, `examples`, `benches`
   - Add comment markers immediately above the relevant public/exported declaration or integration boundary.
   - Put decorators in actual comments. Decorator-like text inside string literals, fixture blobs, or prompt templates will not be counted.
   - Accepted decorator forms:
     `@mvs-feature("name")`
     `@mvs-feature:name`
     `@mvs-protocol("name")`
     `@mvs-protocol:name`
   - These are comment-based markers, not runtime language decorators.

3. Use `@mvs-feature` for meaningful product or runtime capabilities.
   - Tag stable capabilities such as authentication, offline storage, sync, import/export, billing, streaming, plugin support, background jobs, or other user-visible/runtime-relevant features.
   - Do not tag every helper function.
   - Use stable names in snake_case, kebab-case, or dotted form.
   - Prefer one feature tag per capability boundary, not noisy duplicates everywhere.

4. Use `@mvs-protocol` for integration and contract surfaces.
   - Tag public contracts such as HTTP APIs, RPC handlers, CLI command surfaces, SDK entrypoints, extension/host handshakes, queue payload contracts, webhooks, or other machine-to-machine boundaries.
   - Add protocol tags wherever a change would break consumers, extensions, plugins, automation, or external callers.
   - If the project has multiple distinct integration surfaces, tag each with its own stable protocol name.

5. Make sure the public API is visible to the scanner.
   - `mvs-manager` also hashes public/exported API signatures.
   - Member signatures inside class-like scopes are stored with their owning type or module so collisions remain visible in the manifest inventory.
   - Rust signatures are canonicalized from the AST, so inventories and filters use forms like `rust:fn run() -> i32` or `rust:impl-fn HostAdapter::connect(&self, target: &str) -> bool`.
   - TypeScript/JavaScript scanning is parser-backed, so named exports, re-exports, multiline exports, and default exports are part of the contract surface even when line-based regexes would miss them; `scan_policy.ts_export_following` can follow same-workspace relative barrel re-exports when a facade such as `index.ts` is the real contract boundary.
   - Go scanning is parser-backed and tracks exported functions, exported methods, exported named types, exported struct fields, exported embedded struct fields, exported interface methods, embedded interface type elements, exported constants, and exported package `var` declarations; `scan_policy.go_export_following` can expand a rooted `.go` facade file to same-package sibling source files while skipping `_test.go` helpers.
   - Rust facade following can now expand a rooted file such as `src/lib.rs` across same-crate `pub mod` graphs, including nested inline public modules, when `scan_policy.rust_export_following` is set to `public_modules`; this keeps private-module files out of the contract surface and also resolves direct same-crate `pub use` facades from private modules onto the public alias names, including associated inherent methods.
   - Python scanning is parser-backed and tracks public `class` declarations, non-underscore `def` declarations, public `type` aliases, and module-level or class-level constants such as `API_VERSION`, `__all__`, or `Worker.STATUS` without promoting nested local helpers or private class bodies; when `__all__` is parseable it becomes the top-level export boundary, including common alias, unpacking, and `+=` composition patterns built from parseable literals, explicit import re-exports are stored in canonical forms like `python:from auth.core import login as authorize`, same-workspace imported `__all__` aliases or `from ... import *` facades resolve when the upstream module export graph is static and parseable, and `scan_policy.python_export_following` plus `scan_policy.python_module_roots` can pin how aggressively cross-module facade resolution is followed.
   - Java scanning is parser-backed and tracks public types, public fields, interface constants, and public or interface methods while ignoring leading annotations in the stored signature; declared package plus nesting context is preserved in canonical forms like `java:type public class demo.AuthApi`, `java:field public String demo.AuthApi.status`, `java:const public static final String demo.AuthApi.Contract.STATE`, and `java:method public String demo.AuthApi.login(...)`.
   - C# scanning is parser-backed and tracks public types, public fields, public constants, public properties, and public or interface methods while ignoring leading attributes in the stored signature; declared namespace plus nesting context is preserved in canonical forms like `csharp:type public class Demo.AuthApi`, `csharp:field public static readonly string Demo.AuthApi.Version`, `csharp:const public string Demo.AuthApi.STATUS_READY`, `csharp:property public string Demo.AuthApi.DisplayName { get }`, and `csharp:method public static string Demo.AuthApi.Login(...)`.
   - Kotlin scanning is parser-backed and tracks public or default-visible `class`, `interface`, `object`, `fun`, `val`, `var`, and top-level `const val` declarations while skipping `private`, `protected`, and `internal`; declared package plus nesting context is preserved in canonical forms like `kotlin:public class demo.auth.AuthApi`, `kotlin:const val demo.auth.API_VERSION: String`, `kotlin:fun demo.auth.AuthApi.login(...)`, and `kotlin:val demo.auth.AuthApi.token: String`.
   - PHP scanning is parser-backed and tracks top-level functions and constants, classes, interfaces, traits, enums, public properties, public or interface constants, and public or interface methods while treating `#` comments as decorators and ignoring attributes in stored signatures; class and interface members are owner-qualified as `AuthApi.run(...)`, `AuthApi.$token`, and `AuthApi::STATUS_READY`.
   - Ruby scanning is parser-backed and tracks `class`, `module`, public `def`, singleton methods, `class << self` method bodies, public `attr_reader`/`attr_writer`/`attr_accessor` macros, and namespace constants while ignoring heredoc fixtures; `private_constant` hides constants until `public_constant` re-exposes them, `module_function` plus `extend self` promote module singleton exports, `private_class_method` hides singleton methods, member signatures use Ruby owner forms like `Demo::AuthApi#login(...)` and `Demo::AuthApi.connect(...)`, and `scan_policy.ruby_export_following` can disable macro-driven export shaping when the repo wants file-local declarations only.
   - Lua scanning is parser-backed and tracks global `function` declarations and returned module-table exports while treating `--` and long-bracket comments as real decorator locations; when a file returns a module root such as `return Api`, that returned root becomes the explicit runtime export boundary, and `scan_policy.lua_export_following` can require explicit returned roots or disable returned-root following entirely.
   - Swift scanning is parser-backed and tracks `public` and `open` types, functions, properties, and inherited protocol requirements; type and protocol members are stored as owner-qualified signatures like `swift:public func AuthApi.login(...)` and `swift:public var SessionContract.token: ...`.
   - Luau scanning is parser-backed and tracks global `function` declarations, `export type` definitions, and returned module-table exports; when a file returns a module root such as `return Api`, that returned root becomes the explicit runtime export boundary while `export type` remains explicit API.
   - Ensure externally consumed surfaces are actually public/exported in the language’s normal way.
   - Examples:
     - TypeScript/JavaScript: `export function`, `export class`, `export interface`, `export type`, `export enum`, `export const`; `scan_policy.ts_export_following = relative_only` can expand relative `export ... from` barrel facades into the concrete followed signatures they expose
     - Rust: `pub fn`, `pub struct`, `pub enum`, `pub trait`, `pub type`, `pub const`, `pub static`, and `pub` impl methods
     - Go: exported `func` names and exported methods start with an uppercase letter; exported types, exported struct fields, exported embedded struct fields, exported interface methods, embedded interface type elements, exported constants, and exported package `var` declarations are also scanned, and `scan_policy.go_export_following = package_only` can expand a rooted `.go` facade file to the rest of its same-package contract while skipping `_test.go` files
     - Rust facade repos: `scan_policy.rust_export_following = public_modules` can expand a rooted `src/lib.rs` or workspace member crate facade across same-crate `pub mod` files while leaving private-module files out, and it can resolve direct same-crate `pub use` facades from private modules onto the exported alias names
     - Python: public `class` declarations, non-underscore `def` functions, decorated class methods, public `type` aliases, and module-level or class-level constants such as `API_VERSION`, `__all__`, and `Worker.STATUS` are scanned; parseable `__all__` narrows the top-level export surface, including common alias, unpacking, and `+=` composition patterns built from parseable literals, explicit import re-exports are recorded in canonical `from ... import ...` or `import ... as ...` forms, same-workspace imported `__all__` aliases or `from ... import *` facades resolve when the upstream module export graph is static and parseable, and `scan_policy.python_export_following` plus `scan_policy.python_module_roots` can pin resolution to nonstandard package roots or disable cross-file facade following entirely
     - Java: public types, public fields, interface constants, and public or interface methods are scanned, and package plus owning-type context is recorded in type, field, constant, and method signatures
     - C#: public types, public fields, public constants, public properties, and public or interface methods are scanned, and namespace plus owning-type context is recorded in type, field, property, constant, and method signatures
     - Kotlin: public or default-visible `class`, `interface`, `object`, `fun`, `val`, `var`, and top-level `const val` declarations are scanned, and package plus owning-type context is recorded in type, property, constant, and function signatures
     - PHP: top-level `function` and `const`, `class`, `interface`, `trait`, `enum`, public properties, and public or interface constants and methods are scanned, and class/interface members are recorded with their owning type name
     - Ruby: `class`, `module`, public `def`, singleton methods, `class << self` method bodies, public `attr_reader`/`attr_writer`/`attr_accessor` macros, and namespace constants are scanned; `private_constant` hides constants until `public_constant` re-exposes them, `module_function` plus `extend self` promote module singleton exports, `private_class_method` removes hidden singleton methods, member signatures record their owning module or class, and `scan_policy.ruby_export_following` can turn off macro-driven export shaping
     - Lua: global `function` declarations and returned module-table exports are scanned, `return Api`-style module roots narrow runtime exports to that returned root, and `scan_policy.lua_export_following` can require an explicit returned root or disable returned-root following
     - Swift: `public` or `open` `struct`, `class`, `enum`, `protocol`, `extension`, `func`, `var`, and inherited protocol requirement declarations are scanned, and member signatures record their owning type or protocol
     - Luau: global `function` declarations, `export type` definitions, and returned module-table exports are scanned, and `return Api`-style module roots narrow runtime exports to that returned root while `export type` stays explicit API
