You are preparing this project for `mvs-manager`, a multidimensional version manager that tracks:

ARCH.FEAT.PROT-CONT

Your job is to make this repository scannable and verifiable by `mvs-manager` without changing behavior unless required.

Work instructions:

1. Initialize or refresh the manifest.
   - If `mvs.json` does not exist, create it by running:
     `mvs-manager generate --root . --manifest mvs.json --context <context>`
   - Choose a short runtime context that matches the app, for example:
     `cli`, `web`, `server`, `edge`, `edge.mobile`, `desktop`
   - If `mvs.json` already exists, preserve its intent and regenerate it instead of replacing it by hand.

2. Add MVS comment decorators to real source files.
   - `mvs-manager` scans these file types:
     `.rs`, `.ts`, `.tsx`, `.js`, `.jsx`, `.go`, `.py`, `.java`, `.kt`, `.cs`, `.php`, `.rb`, `.swift`, `.lua`, `.luau`
   - It ignores these directories:
     `.git`, `node_modules`, `dist`, `build`, `target`, `vendor`, `.next`, `tests`, `examples`, `benches`
   - Add comment markers immediately above the relevant public/exported declaration or integration boundary.
   - Put decorators in actual comments. Decorator-like text inside string literals, fixture blobs, or prompt templates will not be counted.
   - Accepted decorator forms:
     `@mvs-feature("name")`
     `@mvs-feature:name`
     `@mvs-protocol("name")`
     `@mvs-protocol:name`
   - These are comment-based markers, not runtime language decorators.

3. Use `@mvs-feature` for meaningful product or runtime capabilities.
   - Tag stable capabilities such as authentication, offline storage, sync, import/export, billing, streaming, plugin support, background jobs, or other user-visible/runtime-relevant features.
   - Do not tag every helper function.
   - Use stable names in snake_case, kebab-case, or dotted form.
   - Prefer one feature tag per capability boundary, not noisy duplicates everywhere.

4. Use `@mvs-protocol` for integration and contract surfaces.
   - Tag public contracts such as HTTP APIs, RPC handlers, CLI command surfaces, SDK entrypoints, extension/host handshakes, queue payload contracts, webhooks, or other machine-to-machine boundaries.
   - Add protocol tags wherever a change would break consumers, extensions, plugins, automation, or external callers.
   - If the project has multiple distinct integration surfaces, tag each with its own stable protocol name.

5. Make sure the public API is visible to the scanner.
   - `mvs-manager` also hashes public/exported API signatures.
   - Rust signatures are canonicalized from the AST, so inventories and filters use forms like `rust:fn run() -> i32` or `rust:impl-fn HostAdapter::connect(&self, target: &str) -> bool`.
   - TypeScript/JavaScript scanning is parser-backed, so named exports, re-exports, multiline exports, and default exports are part of the contract surface even when line-based regexes would miss them.
   - Go scanning is parser-backed and tracks exported functions, exported methods, exported named types, exported struct fields, exported embedded struct fields, exported interface methods, embedded interface type elements, exported constants, and exported package `var` declarations.
   - Python scanning is parser-backed and tracks public `class` declarations, non-underscore `def` declarations, public `type` aliases, and module-level or class-level constants such as `API_VERSION`, `__all__`, or `Worker.STATUS` without promoting nested local helpers or private class bodies.
   - Java and C# scanning are parser-backed and track public types and methods while ignoring leading annotations or attributes in the stored signature.
   - Kotlin scanning is parser-backed and tracks public or default-visible `class`, `interface`, `object`, and `fun` declarations while skipping `private`, `protected`, and `internal`.
   - PHP scanning is parser-backed and tracks top-level functions and constants, classes, interfaces, traits, enums, public properties, public or interface constants, and public or interface methods while treating `#` comments as decorators and ignoring attributes in stored signatures.
   - Ruby scanning is parser-backed and tracks `class`, `module`, public `def`, singleton methods, `class << self` method bodies, public `attr_reader`/`attr_writer`/`attr_accessor` macros, and namespace constants while ignoring heredoc fixtures; `private_constant` removes hidden constants, and non-public methods stay out.
   - Lua scanning is parser-backed and tracks global `function` declarations and returned module-table exports while treating `--` and long-bracket comments as real decorator locations.
   - Swift scanning is parser-backed and tracks `public` and `open` types, functions, properties, and inherited protocol requirements.
   - Luau scanning is parser-backed and tracks global `function` declarations, `export type` definitions, and returned module-table exports.
   - Ensure externally consumed surfaces are actually public/exported in the languageâ€™s normal way.
   - Examples:
     - TypeScript/JavaScript: `export function`, `export class`, `export interface`, `export type`, `export enum`, `export const`
     - Rust: `pub fn`, `pub struct`, `pub enum`, `pub trait`, `pub type`, `pub const`, `pub static`, and `pub` impl methods
     - Go: exported `func` names and exported methods start with an uppercase letter; exported types, exported struct fields, exported embedded struct fields, exported interface methods, embedded interface type elements, exported constants, and exported package `var` declarations are also scanned
     - Python: public `class` declarations, non-underscore `def` functions, decorated class methods, public `type` aliases, and module-level or class-level constants such as `API_VERSION`, `__all__`, and `Worker.STATUS` are scanned
     - Java/C#: public types and public methods are scanned
     - Kotlin: public or default-visible `class`, `interface`, `object`, and `fun` declarations are scanned
     - PHP: top-level `function` and `const`, `class`, `interface`, `trait`, `enum`, public properties, and public or interface constants and methods are scanned
     - Ruby: `class`, `module`, public `def`, singleton methods, `class << self` method bodies, public `attr_reader`/`attr_writer`/`attr_accessor` macros, and namespace constants are scanned, while `private_constant` hides constants from the public API inventory
     - Lua: global `function` declarations and returned module-table exports are scanned
     - Swift: `public` or `open` `struct`, `class`, `enum`, `protocol`, `extension`, `func`, `var`, and inherited protocol requirement declarations are scanned
     - Luau: global `function` declarations, `export type` definitions, and returned module-table exports are scanned

   - Update `capabilities` to reflect required/provided runtime features that other manifests may validate against.
   - Update `environment.profiles` and `environment.runtime_constraints` when runtime context or platform constraints matter.

   - Use `scan_policy.public_api_includes` to keep only the declarations that define the supported contract.

7. Use the correct versioning trigger.

   - Protocol tag drift increments `PROT`.
   - Public API signature drift increments `PROT`.
   - AI tool schema hash drift increments `PROT`.
